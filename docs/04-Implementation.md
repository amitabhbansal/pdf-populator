# 04 - Implementation

> Project structure, module responsibilities, configuration, rendering flow, and the error-handling / logging / validation strategies. Engineering decisions rather than syntax — the code itself is the reference.

## 1. Project Structure

```
pdf-populator/
├── config/
│   ├── fields.json              # authored: fields to fill + the page each is on
│   └── mappings/<template>.json # generated: calibrated positions + style, per template
├── input/                       # scanned PDFs + data JSON
├── output/                      # generated: page images (per template) + filled PDFs
├── src/
│   ├── settings.py              # central configuration (single edit point)
│   ├── pdf_to_images.py         # PDF pages → PNG images
│   ├── coordinate_calibrator.py # visual click-to-calibrate tool
│   └── pdf_renderer.py          # validation + overlay rendering
└── docs/
```

## 2. Module Responsibilities

| Module | Responsibility | Key detail |
|---|---|---|
| `settings.py` | Single source of truth for configuration | Paths derived from the PDF name; shared `DPI` makes renderer/calibrator drift impossible |
| `pdf_to_images.py` | Render each PDF page to an image for calibration | Fixed DPI so pixel→point conversion stays correct |
| `coordinate_calibrator.py` | One-time visual calibration | Field-driven page display; saves **PDF points**, not pixels; re-runs merge instead of overwrite |
| `pdf_renderer.py` | Fill the form | Cross-file validation, y-flip at the library boundary, per-field style, run summary |

Each module runs standalone (`python src/<module>.py`) and shares no state except `settings.py` and the JSON files — deliberately simple, testable seams.

## 3. Configuration Files

- **`src/settings.py`** — which PDF, which data file, DPI, baseline nudge, default style. Everything else (image dir, mapping path, output name) is derived from the template name.
- **`config/fields.json`** — authored list: `{ "page": 1, "name": "customer_name" }`. The single declaration of *what* to fill and *where it lives*.
- **`config/mappings/<template>.json`** — generated by the calibrator: per-field `page, x, y, font, size, color` plus a `template` block (name, page size). Stored **per template** so calibrating a new form never overwrites an existing one.
- **Data file** (e.g. `input/sample_data.json`) — flat `field name → value`. The field name is the join key across all three files.

## 4. Coordinate Handling (the core correctness concern)

Three coordinate systems appear in the pipeline; every conversion happens **once, at a boundary**:

1. **Image pixels** (top-left origin, y down) — what the calibrator captures from clicks.
2. **PDF points** (bottom-left origin, y up) — the canonical stored form: `x_pt = x_px·72/DPI`, `y_pt = H − y_px·72/DPI`. Stored points are library-agnostic.
3. **Renderer's library space** — PyMuPDF's `insert_text` uses a top-left origin (verified experimentally, not assumed), so the renderer flips back: `y_top = page_height − y_pt`.

The stored coordinate is the text **baseline**; a configurable `BASELINE_NUDGE` (0.35 × font size) lowers it so the clicked point visually centres the text.

## 5. Rendering Flow

```
load settings → load fields.json + mapping + data
→ validate (cross-check all three; report anything missing)
→ for each mapped field: skip if no data / bad page → convert style + coordinates → insert_text overlay
→ save output PDF → log filled/skipped summary
```

The scanned pages are never modified — text is drawn on top (overlay), keeping the original intact.

## 6. Validation Strategy

The dangerous failure mode is a *silently* incomplete fill. Before rendering, the engine cross-checks:

- field declared in `fields.json` but never calibrated → warning ("will NOT be filled")
- field declared but absent from the data file → warning
- data key with no calibrated position → warning (probable typo)
- empty mapping → hard error, abort

Warnings do not block (partial fills can be intentional); every run ends with an explicit summary: `all 9 field(s) filled` or `7 filled, 2 skipped: date, pan`.

## 7. Error Handling & Logging

- Missing/invalid JSON files → clear `logger.error` and clean exit, no traceback for expected failures.
- Invalid colour or unknown font in config → warn and fall back to defaults rather than abort.
- Non-existent page reference → warn and skip that field.
- `logging` (INFO for placements/summary, WARNING for skips, ERROR for aborts) rather than prints in the renderer, so verbosity is controllable when embedded elsewhere.

## 8. Design Principles Applied

- **Configuration-driven** — the engine has zero form knowledge; forms are added by config only.
- **Convert at boundaries** — pixel→point in the calibrator, point→library-space and hex→RGB in the renderer; the stored data stays canonical.
- **Fail loudly, degrade gracefully** — validation reports every gap; per-field issues skip with warnings instead of crashing the run.
- **Simplest thing that satisfies the requirements** — no OCR, no service framework, no database; each of those was considered and rejected for this scope (see `07-Engineering-Decisions.md`).
